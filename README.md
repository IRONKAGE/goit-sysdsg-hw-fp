# **Фінальний проект з системного дизайну**

## **Опис проекту:**

Вам необхідно спроектувати архітектуру масштабованої платформи електронної комерції, подібної до Amazon або Alibaba. Платформа повинна підтримувати мільйони користувачів, сотні тисяч продавців та мільйони товарів, забезпечуючи високу доступність, надійність та продуктивність.

## **Завдання передбачає:**

1. Аналіз функціональних і нефункціональних вимог.
2. Побудову загальної архітектурної схеми.
3. Вибір технологій для обробки запитів, роботи з даними, зменшення затримок тощо.
4. Визначення точок масштабування та потенційних «вузьких місць».
5. Опис стратегій моніторингу, fault tolerance та disaster recovery.
6. Аргументування обраних рішень і аналіз компромісів.

## **Після виконання проекту ви навчитесь:**

1. Формулювати архітектурні рішення, спираючись на бізнес-вимоги.
2. Обґрунтовано обирати між SQL/NoSQL, кешами, чергами, API та іншими компонентами системи.
3. Аналізувати компроміси між масштабованістю, доступністю та консистентністю (CAP-теорема).
4. Ідентифікувати потенційні точки відмови та планувати стратегії їх мінімізації.
5. Мислити як інженер та дизайнер систем, враховуючи технічні обмеження, ризики й пріоритети продукту.

## **Завдання:**

### **I. Функціональні вимоги:**

1. **Управління користувачами та обліковими записами:**
   - Реєстрація та аутентифікація користувачів (покупців та продавців)
   - Управління профілями користувачів
   - Система ролей та прав доступу
2. **Каталог товарів:**
   - Додавання та управління товарами продавцями
   - Категоризація товарів
   - Пошук та фільтрація товарів
   - Відображення детальної інформації про товар
3. **Кошик та оформлення замовлення:**
   - Додавання товарів до кошика
   - Управління кошиком (зміна кількості, видалення товарів)
   - Процес оформлення замовлення (checkout)
   - Інтеграція з платіжними системами
4. **Управління замовленнями:**
   - Обробка та відстеження статусу замовлень
   - Повідомлення користувачів про зміну статусу
   - Історія замовлень для користувачів
   - Система повернень та відшкодувань
5. **Система відгуків та рейтингів:**
   - Можливість залишати відгуки про товари
   - Рейтингова система для товарів та продавців
   - Модерація відгуків
6. **Рекомендаційна система:**
   - Персоналізовані рекомендації товарів
   - Популярні товари та категорії
   - «Часто купують разом» рекомендації
7. **Аналітика та звітність:**
   - Аналітика продажів для продавців
   - Аналітика поведінки користувачів
   - Бізнес-метрики та KPI

### **II. Нефункціональні вимоги:**

1. **Масштабованість:**
   - Підтримка 10 мільйонів щоденних активних користувачів (DAU)
   - Обробка 100 000+ транзакцій на годину в пікові періоди
   - Каталог з 50+ мільйонів товарів
2. **Продуктивність:**
   - Час відповіді API < 200 мс для 99% запитів
   - Час завантаження сторінки < 2 секунд
   - Швидкий пошук серед мільйонів товарів (< 500 мс)
3. **Доступність:**
   - Цільова доступність системи: 99,99%
   - Мінімальний вплив планових обслуговувань на користувачів
4. **Надійність:**
   - Стійкість до відмов окремих компонентів
   - Запобігання втраті даних про замовлення та платежі
   - Механізми відновлення після збоїв
5. **Інтернаціоналізація:**
   - Підтримка кількох мов
   - Підтримка різних валют та податкових систем
   - Адаптація до регіональних особливостей

### **III. Очікувані результати проекту:**

1. **Системний аналіз та оцінка масштабу:**
   - Визначення ключових метрик системи
   - Back-of-the-envelope розрахунки для оцінки необхідних ресурсів
   - Аналіз потенційних вузьких місць
2. **Високорівнева архітектура:**
   - Діаграма архітектури системи
   - Опис основних компонентів та їх взаємодії (не менше 5 компонентів)
   - Обґрунтування вибору архітектурного підходу
3. **Детальний дизайн ключових компонентів:**
   - Дизайн API (покрити основні флоу — від 4 інтерфейсів)
   - Схеми баз даних (від 4 сутностей)
   - Комунікація між компонентами
4. **Стратегії забезпечення надійності та відмовостійкості:**
   - Реплікація та шардування даних
   - Механізми обробки відмов
   - Стратегії відновлення після збоїв
5. **Підходи до масштабування:**
   - Горизонтальне та вертикальне масштабування
   - Стратегії для обробки пікових навантажень (наприклад, під час розпродажів)
   - Оптимізація продуктивності
6. **Аналіз компромісів:**
   - Обґрунтування архітектурних рішень
   - Аналіз компромісів між різними властивостями системи
   - Оцінка альтернативних підходів (розгляд 1-2 альтернатив для кожного основного компонента системи)

### **IV. Додаткові вказівки:**

1. Проект має бути представлений у вигляді документа з чіткою структурою, що відповідає очікуваним результатам. Ви можете додати зміст на першу сторінку документа.
2. (Це опціонально, але рекомендую їх додати). Використовуйте діаграми, схеми та інші візуальні елементи для ілюстрації архітектурних рішень.
3. Обґрунтовуйте всі ключові рішення.
4. Розглядайте систему в контексті реальних умов експлуатації, враховуючи обмеження ресурсів, вартість та складність підтримки.
5. Приділіть особливу увагу механізмам забезпечення узгодженості даних, особливо в контексті розподілених транзакцій (наприклад, оновлення запасів товарів при оформленні замовлень).
6. Розгляньте стратегії для обробки надзвичайно високих навантажень під час спеціальних подій, таких як «Чорна п'ятниця» або великі розпродажі.

### ***V. Тож ваші прикріплені файли або один файл має містити:***

1. Якщо обрали не систему електронної комерції – обов’язково вкажіть яку і зазначте функціональні, а також нефункціональні вимоги й способи їх забезпечення
2. Стислий опис функціональності системи й основні бізнес-вимоги
3. Архітектурну схему системи з поясненням компонентів та взаємодії між ними
4. Вибір і обґрунтування рішень щодо:
   - типу сховища (SQL / NoSQL);
   - стратегії кешування;
   - масштабування (горизонтальне / вертикальне);
   - способу комунікації між сервісами (синхронна / асинхронна);
   - обробки черг і подій (за потреби)
5. Якщо обрано систему електронної комерції — нефункціональних вимог (надійність, продуктивність, доступність) і способи їх забезпечення
6. Опис fault-tolerance-рішень: як система поводиться у випадку відмови компонентів
7. Опис стратегії моніторингу та логування
8. Потенційні проблеми, ризики та способи їх пом’якшення
9. Обґрунтування прийнятих технічних рішень (чому саме так, а не інакше)
10. (*Опціонально*) Діаграми або схеми для кращої візуалізації — блок-схеми, послідовності, топології тощо.
